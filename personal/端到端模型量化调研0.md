# 端到端模型量化调研（初稿 0）

本文汇总对话中围绕 **SparseDrive 类端到端自动驾驶模型** 的参数量讨论、**量产侧量化** 的文献与公开资料可引用性，以及**表述边界**（哪些是工程归纳、哪些有明确出处）。便于后续写报告或论文时选用措辞。

---

## 1. SparseDrive 类模型参数量（量级）

### 1.1 公开信息

- 官方 **README / 论文未给出「总参数量」** 的单一数字。
- 结构要点（以仓库 `projects/configs/sparsedrive_small_stage1.py` 等为例）：
  - **SparseDrive-S**：`ResNet50` 骨干 + FPN + 深度辅助分支 + 对称稀疏感知（`embed_dims=256`、多层 decoder 等）+ 并行运动/规划头。
  - **SparseDrive-B**（论文）：**`ResNet101` + 更大输入分辨率**（如 512×1408），整体更重。

### 1.2 经验推断（非官方数字）

- 仅骨干：**ResNet50 ≈ 25M、ResNet101 ≈ 44M** 量级（通用文献/实现中的常见规模）。
- 其余模块（可变形聚合、多任务头、时序等）通常再占 **千万～一亿级** 不等。
- **整体粗估**：**约几千万～一亿多参数（约 0.05B～0.12B 量级）** 对 SparseDrive 这类「稀疏、省 BEV」栈较合理；**多数配置会低于 0.2B（2 亿）**。
- **0.2B** 更常出现在**更重的一体化 BEV/大栈**讨论中，**不宜**在未经本机 `model.parameters()` 或官方统计的情况下，把 SparseDrive 某变体**断言**为 0.2B。

### 1.3 若需精确参数量

- 用配置 `build` 模型后对 `sum(p.numel() for p in model.parameters())` 统计；或参考仓库 `tools/benchmark.py` 中与 FLOPs/参数量相关的逻辑（依赖环境与数据加载是否可用）。

**论文链接**：<https://arxiv.org/abs/2405.19620>

## 2. 「训练精度」：论文 / 开源常见做法

- 大规模视觉与检测管线普遍使用 **FP32 + 混合精度（FP16/bfloat16）** 训练；MM 系配置里常见 `fp16` 等项。
- 本仓库配置示例中含 `fp16 = dict(loss_scale=32.0)`，属训练侧混合精度常见写法。

**说明**：这属于**领域通例**，一般不必为「是否用 FP16 训练」单独找一篇论文证成。

## 3. 「部署量化」：可引用公开资料（偏感知与推理引擎）

### 3.1 车载端对深度网络做 INT8 推理（含感知类网络）

- **NVIDIA TensorRT** 与 **DRIVE 平台** 上，对语义分割等网络做 INT8 校准与部署的技术博文（含与 FP32 优化对比、INT8 进一步加速等）。  
  - 链接：<https://developer.nvidia.com/blog/fast-int8-inference-for-autonomous-vehicles-with-tensorrt-3/>  
  - **注意**：文章有年代；**结论性**可写「行业长期存在用 TensorRT/INT8 做车载 DNN 推理优化的实践」，具体倍数与硬件以当前文档为准。

### 3.2 开源自动驾驶栈中的感知侧 INT8 案例

- **Autoware Lite Models**：对车道线分割、场景分割、单目深度等**感知任务**做 TensorRT INT8、面向嵌入式（如 Jetson Orin Nano）的优化说明。  
  - 链接：<https://autoware.org/introducing-lite-models/>

以上两类支撑：**「感知类深度网络在工程上广泛采用 INT8/TensorRT 等部署路径」**。

## 4. 策略 / 序贯任务对量化更敏感（论文支撑「决策链要谨慎」）

下列文献**不**等同于「某 OEM 量产栈里规划头固定 FP16」，但支撑机制：**整网激进低比特可能影响转向/安全相关行为，序贯任务与单次前向的感知不同**。

- **Park et al., *Quantization-Aware Imitation-Learning for Resource-Efficient Robotic Control***（arXiv:2412.01034，含产业合作作者）  
  - 讨论自动驾驶中量化对**转向等决策**的影响及碰撞风险示意；  
  - 对比 **QAT 在典型感知任务（单次推理）** 与 **序贯决策任务** 上的差异。  
  - 链接：<https://arxiv.org/abs/2412.01034>

### 4.1 感知任务量化的补充文献（研究侧）

- **PTQAT** 等面向 **3D 感知** 的混合 PTQ/QAT 工作（如 ICCV 2025 Workshop：*PTQAT: A Hybrid Parameter-Efficient Quantization Algorithm for 3D Perception Tasks*），说明公开研究里**感知 + 量化**的系统工作较多。  
  - 可在 OpenAccess 按题名检索 PDF。

## 5. 关于「感知/backbone 先量化、规划子网保守处理」这句话

### 5.1 性质说明

- 该表述是对**工程实践与风险分层**的**归纳**，**不是**某篇论文或白皮书的**逐字原文**。
- **Tier1/OEM 的完整流水线拆分**（哪一层 INT8、哪一层 FP16）多数**未公开**，不宜写成全行业统一比例。

### 5.2 可写进正式材料的稳妥表述建议

- **可证**：公开资料常见 **感知 DNN + TensorRT INT8**；学术工作指出 **policy/序贯任务** 对量化更敏感、需 QAT/混合精度等。  
- **慎写**：在未引用具体项目的前提下，**不要**写死「必然先量化 backbone、规划一定全精度」等**无法举证**的细节。
