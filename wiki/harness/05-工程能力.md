# 5. 工程能力

## 5.1 Harness 架构

### 5.1.1 测试框架设计

- **名称**：模型 Harness 分层测试框架
- **简介**：围绕数据加载、模型前处理、推理、后处理、指标计算和报告输出构建统一入口。
- **适用场景**：自动驾驶感知模型离线回放、训练后验收、部署前回归、跨模型横向评估。
- **优势或局限**：可复用测试逻辑并提升可追溯性；局限是需要稳定的数据 schema、指标定义和环境封装。
- **参考资料链接**：
  - https://github.com/open-mmlab/OpenPCDet
  - https://github.com/nutonomy/nuscenes-devkit
  - https://github.com/tier4/autoware-ml

### 5.1.2 Benchmark 测试

- **名称**：Benchmark 基准测试
- **简介**：用固定数据集、固定模型配置和固定硬件环境测量精度、延迟、吞吐、显存和稳定性。
- **适用场景**：模型选型、TensorRT/ONNX 导出验收、量化对比、版本发布门禁。
- **优势或局限**：便于横向比较和回归发现；局限是基准场景可能不能完全覆盖真实长尾工况。
- **参考资料链接**：
  - https://mlcommons.org/benchmarks/inference-datacenter/
  - https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/perf_analyzer/README.html
  - https://github.com/open-mmlab/mmdetection3d

### 5.1.3 自动化测试工具

- **名称**：pytest / GitHub Actions / GitLab CI / Jenkins
- **简介**：把单元测试、集成测试、模型 smoke test、导出测试和基准测试串成自动化流水线。
- **适用场景**：训练代码合入、模型导出脚本变更、数据 schema 变更、部署镜像更新。
- **优势或局限**：降低人工验收成本；局限是 GPU CI 成本较高，长耗时测试需要分层调度。
- **参考资料链接**：
  - https://docs.pytest.org/
  - https://docs.github.com/actions

## 5.2 性能分析与优化

### 5.2.1 GPU/CPU/NPU 性能分析

- **名称**：异构硬件性能分析
- **简介**：从算子耗时、内存带宽、数据搬运、线程调度、batch 策略和硬件利用率定位瓶颈。
- **适用场景**：自动驾驶车端低延迟推理、训练吞吐优化、边缘 NPU 部署和多传感器 pipeline 优化。
- **优势或局限**：能量化瓶颈并指导优化优先级；局限是不同硬件工具链差异大，结果需要和业务指标联动解释。
- **参考资料链接**：
  - https://docs.nvidia.com/datacenter/dcgm/latest/
  - https://developer.nvidia.com/nsight-systems
  - https://www.intel.com/content/www/us/en/developer/tools/vtune-profiler/overview.html

### 5.2.2 Profiling 工具（Nsight、PyTorch Profiler）

- **名称**：Nsight Systems / Nsight Compute / PyTorch Profiler
- **简介**：Nsight 分析 CUDA timeline、kernel 和系统级调用；PyTorch Profiler 追踪 eager/compiled 模型的算子与内存。
- **适用场景**：定位 GPU 空洞、CPU dataloader 瓶颈、kernel 低效、同步点过多和模型推理延迟抖动。
- **优势或局限**：能提供从 Python 到 CUDA kernel 的证据链；局限是 profile 会引入额外开销，采样窗口要控制。
- **参考资料链接**：
  - https://docs.nvidia.com/nsight-systems/
  - https://docs.nvidia.com/nsight-compute/
  - https://pytorch.org/tutorials/recipes/recipes/profiler_recipe.html
  - https://docs.nvidia.com/nsight-operator/

### 5.2.3 推理优化技巧

- **名称**：图优化、量化、batching、算子融合与内存复用
- **简介**：通过 ONNX/TensorRT/OpenVINO 导出、FP16/INT8、动态 batch、CUDA Graph、缓存和异步 pipeline 提升推理效率。
- **适用场景**：感知模型上线、车端多模型并发、云端大规模回放和仿真评估。
- **优势或局限**：能显著降低延迟和资源成本；局限是精度校准、动态 shape 和自定义算子可能增加工程复杂度。

## 5.3 工程方法论

### 5.3.1 模块化工程

- **名称**：配置、数据、模型、评估、部署解耦
- **简介**：用清晰边界管理训练配置、数据适配器、模型结构、指标评估、导出部署和测试 harness。
- **适用场景**：多团队协作、多模型复用、跨数据集实验、车端/云端双部署。
- **优势或局限**：提高复用和可维护性；局限是早期需要投入接口设计和治理成本。
- **参考资料链接**：
  - https://hydra.cc/docs/intro/
  - https://lightning.ai/docs/pytorch/stable/
  - https://github.com/open-mmlab/mmengine

### 5.3.2 可复现实验

- **名称**：实验可复现规范
- **简介**：记录代码 commit、数据版本、配置、随机种子、依赖镜像、硬件信息和训练日志。
- **适用场景**：模型效果复盘、bug 定位、审计合规、论文/报告复现和模型发布验收。
- **优势或局限**：降低“同配置不同结果”的排查成本；局限是完全确定性在 GPU 并行训练中可能牺牲性能。
- **参考资料链接**：

### 5.3.3 自动化数据/模型管线

- **名称**：自动化数据与模型流水线
- **简介**：把数据抽取、清洗、标注、训练、评估、注册、部署和监控组织成可追踪 DAG。
