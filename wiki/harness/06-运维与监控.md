## 6.1 日志与指标

### 6.1.1 日志收集工具（ELK、Fluentd）

- **名称**：Elastic Stack（Elasticsearch、Logstash、Kibana）
  - **简介**：Elastic Stack 提供日志采集、索引、搜索和可视化能力，常与 Beats/Logstash 组成集中式日志平台。
  - **适用场景**：训练任务日志检索、推理服务错误追踪、CI/CD 运行历史审计、车端回传日志分析。
  - **优势或局限**：查询和可视化能力强；大规模索引需要容量规划、冷热分层和保留策略。
  - **参考资料链接**：https://www.elastic.co/elastic-stack/
- **名称**：Fluentd / Fluent Bit
  - **简介**：Fluentd 是日志收集与转发器，Fluent Bit 更轻量，适合容器和边缘节点。
  - **适用场景**：Kubernetes Pod 日志采集、车端边缘设备日志转发、训练集群统一日志入口。
  - **优势或局限**：插件生态成熟、资源占用可控；复杂解析规则需要严谨测试，避免丢字段或重复写入。
  - **参考资料链接**：https://docs.fluentd.org/ ，https://docs.fluentbit.io/
- **名称**：OpenTelemetry Logs
  - **简介**：OpenTelemetry 提供日志、指标、链路追踪的统一采集标准。
  - **适用场景**：把训练 harness、数据服务、模型服务的日志与 trace 统一关联。
  - **优势或局限**：标准化程度高，利于跨供应商迁移；日志信号仍在快速演进，落地时需确认 Collector 与后端兼容性。
  - **参考资料链接**：https://opentelemetry.io/docs/concepts/signals/logs/
- **名称**：Loki
  - **简介**：Grafana Loki 是面向标签索引的日志系统，与 Prometheus/Grafana 体验接近。
  - **适用场景**：Kubernetes 日志、轻量训练平台日志、需要按标签筛选而非全文索引的场景。
  - **优势或局限**：存储成本较低、与 Grafana 集成顺滑；复杂全文搜索能力不如 Elasticsearch。
  - **参考资料链接**：https://grafana.com/docs/loki/latest/
- **名称**：结构化日志规范
  - **简介**：对日志字段、级别、请求 ID、模型版本、数据版本、场景 ID 进行统一约束。
  - **适用场景**：自动驾驶模型训练失败诊断、线上推理异常定位、回放任务审计。
  - **优势或局限**：提升检索和关联能力；需要 SDK、代码审查和 lint 约束持续维护。
  - **参考资料链接**：https://opentelemetry.io/docs/specs/otel/logs/

### 6.1.2 指标监控（Prometheus、Grafana）

- **名称**：Prometheus
  - **简介**：Prometheus 是云原生指标采集、存储和告警系统，使用 PromQL 查询时间序列。
  - **适用场景**：训练吞吐、GPU 利用率、推理延迟、队列长度、错误率和资源饱和度监控。
  - **优势或局限**：Kubernetes 生态成熟；高基数标签容易导致存储和查询压力。
  - **参考资料链接**：https://prometheus.io/docs/introduction/overview/
- **名称**：Grafana
  - **简介**：Grafana 提供跨数据源仪表盘、告警和探索能力。
  - **适用场景**：模型服务 SLO 面板、GPU 集群资源面板、训练实验健康度面板。
  - **优势或局限**：可视化能力强、模板丰富；仪表盘需要治理，避免指标定义漂移。
  - **参考资料链接**：https://grafana.com/docs/grafana/latest/
- **名称**：NVIDIA DCGM Exporter
  - **简介**：DCGM Exporter 暴露 NVIDIA GPU 指标到 Prometheus。
  - **适用场景**：GPU 利用率、显存、温度、功耗、XID 错误监控。
  - **优势或局限**：GPU 指标覆盖全面；需要与驱动、DCGM、Kubernetes GPU Operator 版本匹配。
  - **参考资料链接**：https://github.com/NVIDIA/dcgm-exporter
- **名称**：KServe / Triton 指标面板
  - **简介**：KServe 提供模型服务 Grafana Dashboard，Triton 暴露推理请求、队列、计算和 GPU 指标。
  - **适用场景**：在线推理延迟拆解、批处理策略评估、模型版本发布观察。
  - **优势或局限**：与模型服务运行时贴合；跨服务端到端 tracing 仍需额外接入。
  - **参考资料链接**：https://kserve.github.io/website/docs/model-serving/predictive-inference/observability/grafana-dashboards ，https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/metrics.html
- **名称**：告警规则设计
  - **简介**：围绕可用性、延迟、错误率、数据积压、GPU 饱和度建立分层告警。
  - **适用场景**：训练平台值班、推理服务 SLO、数据管线 SLA。
  - **优势或局限**：能缩短故障发现时间；阈值过细会导致告警疲劳，需要分级和抑制策略。
  - **参考资料链接**：https://prometheus.io/docs/alerting/latest/overview/

## 6.2 容器化与资源管理

### 6.2.1 Docker 容器化

- **名称**：Docker
  - **简介**：Docker 通过镜像和容器封装运行环境，是训练、评测、部署一致性的基础。
  - **适用场景**：CUDA/PyTorch 环境固化、CI 测试、推理服务镜像、数据处理任务镜像。
  - **优势或局限**：可移植性好；GPU、驱动、系统库和镜像层管理需要标准化。
  - **参考资料链接**：https://docs.docker.com/
- **名称**：多阶段构建
  - **简介**：在 Dockerfile 中分离构建阶段和运行阶段，减少最终镜像体积。
  - **适用场景**：编译自定义 CUDA 算子、构建 C++ 预处理工具、打包 Python wheel。
  - **优势或局限**：镜像更小、更安全；Dockerfile 可读性和缓存策略需要维护。
  - **参考资料链接**：https://docs.docker.com/build/building/multi-stage/
- **名称**：NVIDIA Container Toolkit
  - **简介**：NVIDIA Container Toolkit 让容器访问宿主机 GPU。
  - **适用场景**：GPU 训练容器、Triton 容器、profiling 容器、边缘推理容器。
  - **优势或局限**：GPU 容器运行标准方案；依赖宿主机驱动和容器运行时配置。
  - **参考资料链接**：https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/latest/
- **名称**：镜像安全扫描
  - **简介**：对基础镜像、系统包、Python 包和模型服务镜像进行漏洞扫描。
  - **适用场景**：生产推理镜像发布、训练平台基础镜像更新、第三方工具镜像准入。
  - **优势或局限**：降低供应链风险；误报和 CVE 修复窗口需要安全策略管理。
  - **参考资料链接**：https://docs.docker.com/scout/
- **名称**：可复现镜像规范
  - **简介**：固定基础镜像 digest、依赖版本、构建参数和启动入口。
  - **适用场景**：训练结果复现、回归 benchmark、线上模型回滚。
  - **优势或局限**：显著降低环境漂移；依赖升级需要通过自动化验证。
  - **参考资料链接**：https://docs.docker.com/build/reproducible-builds/

### 6.2.2 Kubernetes / K8s 集群管理

- **名称**：Kubernetes
  - **简介**：Kubernetes 提供容器编排、调度、服务发现、扩缩容和声明式资源管理。
  - **适用场景**：训练任务调度、推理服务部署、数据处理流水线、GPU 集群管理。
  - **优势或局限**：生态成熟、可扩展性强；学习曲线陡，GPU/存储/网络配置复杂。
  - **参考资料链接**：https://kubernetes.io/docs/home/
- **名称**：Kueue
  - **简介**：Kueue 是 Kubernetes 原生作业排队系统，面向批处理和 AI/ML 工作负载。
  - **适用场景**：共享 GPU 集群上的训练作业排队、配额管理、优先级调度。
  - **优势或局限**：适合多团队资源治理；需要与 Job、RayJob、MPIJob 等工作负载控制器配合。
  - **参考资料链接**：https://kueue.sigs.k8s.io/
- **名称**：Kubeflow Training Operator
  - **简介**：Training Operator 支持 PyTorchJob、TFJob、MPIJob 等分布式训练 CRD。
  - **适用场景**：在 K8s 上运行 DDP、Horovod、TensorFlow 分布式训练。
  - **优势或局限**：贴合 ML 作业生命周期；不同框架的错误恢复和日志收集需要额外工程化。
  - **参考资料链接**：https://www.kubeflow.org/docs/components/training/
- **名称**：Kubernetes HPA / KEDA
  - **简介**：HPA 根据指标扩缩容，KEDA 支持事件驱动扩缩容。
  - **适用场景**：推理服务按 QPS、队列长度、Kafka backlog 扩缩容。
  - **优势或局限**：提升资源利用率；模型冷启动和 GPU 粒度扩容可能影响延迟。
  - **参考资料链接**：https://keda.sh/docs/latest/
- **名称**：资源隔离与配额
  - **简介**：通过 Namespace、ResourceQuota、LimitRange、PriorityClass 管理多团队资源。
  - **适用场景**：训练平台多租户、推理服务等级隔离、共享 GPU 池治理。
  - **优势或局限**：防止资源抢占失控；配额过硬会降低集群整体利用率。
  - **参考资料链接**：https://kubernetes.io/docs/concepts/policy/resource-quotas/

### 6.2.3 NVIDIA NGC 镜像与 GPU 调度

- **名称**：NVIDIA NGC Catalog
  - **简介**：NGC 提供 NVIDIA 优化的框架容器、模型、Helm Chart 和资源。
  - **适用场景**：快速获取 PyTorch/TensorFlow/Triton/Nsight 等 GPU 优化镜像。

  - **参考资料链接**：https://catalog.ngc.nvidia.com/
- **名称**：NVIDIA GPU Operator
  - **简介**：GPU Operator 自动部署驱动、Container Toolkit、Device Plugin、DCGM 等组件。
  - **适用场景**：Kubernetes GPU 集群初始化、升级和监控基础组件管理。
  - **优势或局限**：降低 GPU 集群运维复杂度；依赖节点 OS、内核、驱动兼容性。
  - **参考资料链接**：https://docs.nvidia.com/datacenter/cloud-native/gpu-operator/latest/
- **名称**：NVIDIA Kubernetes Device Plugin
  - **简介**：Device Plugin 将 GPU 资源暴露给 Kubernetes 调度器。
  - **适用场景**：训练 Pod、推理 Pod、profiling Pod 的 GPU 资源申请。
  - **优势或局限**：K8s GPU 调度基础组件；默认粒度为整卡，切分需 MIG 或其他方案。
  - **参考资料链接**：https://github.com/NVIDIA/k8s-device-plugin
- **名称**：MIG（Multi-Instance GPU）
  - **简介**：MIG 可将部分 NVIDIA GPU 切分为多个隔离实例。
  - **适用场景**：小模型推理、多租户实验、资源隔离要求高的服务。
  - **优势或局限**：提高 GPU 利用率和隔离性；只支持特定 GPU，配置和监控更复杂。
  - **参考资料链接**：https://docs.nvidia.com/datacenter/tesla/mig-user-guide/
- **名称**：GPU 调度策略
  - **简介**：结合节点标签、taint/toleration、亲和性、拓扑感知和配额进行调度。
  - **适用场景**：大规模 DDP 训练、低延迟推理、数据本地性敏感作业。
  - **优势或局限**：能提升吞吐和稳定性；策略过多会导致排队时间增长和资源碎片。
  - **参考资料链接**：https://kubernetes.io/docs/concepts/scheduling-eviction/assign-pod-node/

## 6.3 故障诊断

### 6.3.1 异常检测

- **名称**：训练异常检测
  - **简介**：监控 loss、梯度范数、学习率、吞吐、NaN/Inf、数据加载失败等异常。
  - **适用场景**：长周期训练、分布式训练、自动化实验批量运行。
  - **优势或局限**：能尽早终止无效实验；阈值依赖任务和模型，需要结合历史基线。
  - **参考资料链接**：https://docs.pytorch.org/docs/stable/notes/amp_examples.html
- **名称**：推理异常检测
  - **简介**：监控输入分布漂移、输出置信度异常、延迟尖刺、错误率和资源饱和。
  - **适用场景**：线上模型服务、边缘推理、回放评测服务。
  - **优势或局限**：帮助发现数据漂移和服务退化；误报需要与业务场景和标注反馈闭环校准。
  - **参考资料链接**：https://docs.evidentlyai.com/
- **名称**：数据管线异常检测
  - **简介**：检测缺帧、时间戳跳变、标定参数异常、schema 变化和数据体积异常。
  - **适用场景**：自动驾驶日志入湖、样本挖掘、训练集构建。
  - **优势或局限**：能阻断脏数据进入训练；需要清晰的数据契约和质量阈值。
  - **参考资料链接**：https://docs.greatexpectations.io/
- **名称**：GPU 故障异常
  - **简介**：关注 XID 错误、ECC 错误、显存泄漏、温度和功耗异常。
  - **适用场景**：训练集群运维、推理集群 SRE、性能 benchmark 异常定位。
  - **优势或局限**：能快速识别硬件和驱动问题；日志需要与节点事件、Pod 事件关联。
  - **参考资料链接**：https://docs.nvidia.com/deploy/xid-errors/
- **名称**：异常分级与自动处置
  - **简介**：按影响范围和可恢复性定义重试、降级、隔离、熔断和告警升级。
  - **适用场景**：自动训练平台、模型服务平台、离线数据处理平台。
  - **优势或局限**：降低人工值班压力；自动处置策略需要防止反复重启和故障放大。
  - **参考资料链接**：https://sre.google/sre-book/monitoring-distributed-systems/

### 6.3.2 日志分析与报警

- **名称**：关联 ID 追踪
  - **简介**：在请求、任务、样本、模型版本、数据版本之间传递 trace_id/run_id。
  - **适用场景**：从线上误检回溯到样本、训练 run、模型产物和部署版本。
  - **优势或局限**：根因定位效率高；需要全链路字段规范和中间件支持。
  - **参考资料链接**：https://opentelemetry.io/docs/concepts/signals/traces/
- **名称**：日志告警
  - **简介**：基于错误关键字、结构化字段、错误率和异常模式触发告警。
  - **适用场景**：训练失败、推理服务异常、数据导入失败、GPU XID 错误。
  - **优势或局限**：覆盖指标难以捕捉的语义错误；关键字规则容易误报，需要结构化字段辅助。
  - **参考资料链接**：https://grafana.com/docs/loki/latest/alert/
- **名称**：Runbook
  - **简介**：把常见报警的判断步骤、命令、仪表盘和恢复动作文档化。
  - **适用场景**：训练任务挂起、推理延迟升高、数据积压、节点 GPU 异常。
  - **优势或局限**：降低值班门槛；需要随系统演进持续更新。
  - **参考资料链接**：https://sre.google/workbook/alerting-on-slos/
- **名称**：根因分析报告
  - **简介**：记录时间线、影响范围、直接原因、根因、修复项和预防措施。
  - **适用场景**：线上事故、训练平台大面积失败、数据污染事件。
  - **优势或局限**：推动系统性改进；需要无责文化和明确 action owner。
  - **参考资料链接**：https://sre.google/sre-book/postmortem-culture/
- **名称**：告警降噪
  - **简介**：通过分组、抑制、静默、SLO burn rate 和动态阈值减少噪声。
  - **适用场景**：多模型多集群推理平台、训练平台批量失败场景。
  - **优势或局限**：提升告警可信度；规则过度抑制可能漏报真实故障。
  - **参考资料链接**：https://prometheus.io/docs/alerting/latest/alertmanager/

### 6.3.3 系统优化

- **名称**：容量规划
  - **简介**：基于训练排队、GPU 利用率、推理峰值流量和存储增长预测容量。
  - **适用场景**：年度 GPU 采购、云资源预算、推理服务扩容、数据湖存储规划。
  - **优势或局限**：降低资源短缺和浪费；预测依赖历史负载和业务节奏。
  - **参考资料链接**：https://sre.google/sre-book/software-engineering-in-sre/
- **名称**：成本优化
  - **简介**：通过混部、spot/preemptible、作业优先级、模型压缩和缓存降低成本。
  - **适用场景**：离线训练、批量推理、数据预处理、低优先级实验。
  - **优势或局限**：节省明显；需要处理抢占、重试和结果一致性。
  - **参考资料链接**：https://kueue.sigs.k8s.io/docs/concepts/cluster_queue/
- **名称**：可靠性优化
