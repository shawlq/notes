# 1. 模型训练与推理框架

## 1.1 深度学习框架

### 1.1.1 PyTorch

- **名称**：PyTorch
- **简介**：动态图深度学习框架，生态覆盖训练、分布式、导出、编译优化与模型服务，是自动驾驶感知和端到端训练中最常见的研究与工程框架。
- **适用场景**：多传感器感知模型训练、BEV/Transformer 网络研发、在线实验迭代、DDP/FSDP 多 GPU 训练、ONNX/TensorRT 导出前的基准模型实现。
- **优势或局限**：优势是调试友好、社区活跃、与 CUDA/NVIDIA 生态集成深；局限是动态图模型在生产推理前通常需要导出、编译或重写部分算子。
- **参考资料链接**：https://pytorch.org/docs/stable/index.html；https://docs.pytorch.org/tutorials/recipes/recipes/tuning_guide.html；https://docs.pytorch.org/tutorials/beginner/dist_overview.html

- **名称**：torch.compile / PyTorch 2.x 编译栈
- **简介**：PyTorch 2.x 通过 Dynamo、AOTAutograd、Inductor 等组件将部分动态图捕获为可优化图，减少 Python 调度开销。
- **适用场景**：训练吞吐优化、推理前快速验证 kernel fusion 收益、对固定 shape 或有限动态 shape 的感知模型做性能探索。
- **优势或局限**：优势是对原 PyTorch 代码侵入较小；局限是自定义 CUDA 扩展、复杂控制流和高度动态输入可能触发 graph break。
- **参考资料链接**：https://pytorch.org/get-started/pytorch-2.0/；https://docs.pytorch.org/docs/stable/generated/torch.compile.html

### 1.1.2 TensorFlow

- **名称**：TensorFlow / Keras
- **简介**：端到端机器学习框架，包含 Keras 高层 API、SavedModel、TensorBoard、TF Serving 和 TFLite 等组件。
- **适用场景**：已有 TensorFlow 资产维护、移动端/嵌入式部署、TensorBoard 训练监控、需要 SavedModel 标准格式的服务化流程。
- **优势或局限**：优势是部署链路成熟、移动端和生产工具丰富；局限是在自动驾驶新研究代码中热度低于 PyTorch，跨框架迁移需要额外校验。
- **参考资料链接**：https://www.tensorflow.org/guide；https://keras.io/guides/；https://www.tensorflow.org/tensorboard

- **名称**：TensorFlow Lite
- **简介**：面向移动端和边缘设备的轻量推理运行时，支持量化和硬件委托。
- **适用场景**：车端辅助模块、低功耗边缘验证、Android 工具链集成和非 NVIDIA 硬件的原型部署。
- **优势或局限**：优势是轻量、跨平台；局限是复杂 3D 感知算子和动态 shape 支持常需要替代实现或模型改写。
- **参考资料链接**：https://www.tensorflow.org/lite；https://www.tensorflow.org/lite/performance/model_optimization

✅ 本次抓取完成（页 2/61，维度 1.1.2 TensorFlow）

### 1.1.3 JAX

- **名称**：JAX
- **简介**：基于 XLA 的数组计算与自动微分框架，核心能力包括 `jit`、`vmap`、`pmap` 和函数式随机数管理。
- **适用场景**：大规模研究实验、可组合自动微分、仿真/控制优化、需要 TPU 或 XLA 编译优化的训练任务。
- **优势或局限**：优势是编译优化和函数变换能力强；局限是生态中自动驾驶现成模型与部署工具少于 PyTorch。
- **参考资料链接**：https://jax.readthedocs.io/en/latest/；https://jax.readthedocs.io/en/latest/notebooks/quickstart.html

- **名称**：Flax / Orbax
- **简介**：Flax 是 JAX 神经网络库，Orbax 提供检查点保存与恢复能力。
- **适用场景**：JAX 研究模型工程化、分布式训练 checkpoint、函数式模块管理。
- **优势或局限**：优势是与 JAX 原生设计一致；局限是团队需要熟悉纯函数式状态管理，迁移 PyTorch 代码成本较高。
- **参考资料链接**：https://flax.readthedocs.io/；https://orbax.readthedocs.io/

✅ 本次抓取完成（页 3/61，维度 1.1.3 JAX）

### 1.1.4 ONNX

- **名称**：ONNX
- **简介**：开放神经网络交换格式，用图结构描述模型算子、张量类型和 shape，是训练框架到推理引擎之间的常用中间格式。
- **适用场景**：PyTorch/TensorFlow 到 TensorRT、OpenVINO、ONNX Runtime 的模型导出，跨平台回归测试，推理引擎兼容性检查。
- **优势或局限**：优势是生态中立、便于跨工具流转；局限是部分自定义算子、动态控制流和新算子集需要版本锁定或插件实现。
- **参考资料链接**：https://onnx.ai/；https://onnx.ai/onnx/intro/；https://onnxruntime.ai/docs/

- **名称**：ONNX Runtime
- **简介**：高性能推理运行时，支持 CPU、CUDA、TensorRT、OpenVINO、DirectML 等 execution provider。
- **适用场景**：离线回归、跨硬件基线性能测试、非 NVIDIA 环境部署、模型导出正确性验证。
- **优势或局限**：优势是平台覆盖广、性能基线稳定；局限是极致 NVIDIA GPU 性能通常仍需 TensorRT 专项优化。
- **参考资料链接**：https://onnxruntime.ai/docs/execution-providers/；https://onnxruntime.ai/docs/performance/

✅ 本次抓取完成（页 4/61，维度 1.1.4 ONNX）

## 1.2 推理优化工具

### 1.2.1 TensorRT

- **名称**：NVIDIA TensorRT
- **简介**：NVIDIA GPU 上的深度学习推理优化 SDK，支持图优化、层融合、kernel auto-tuning、FP16/INT8 量化和动态 shape profile。
- **适用场景**：车端 NVIDIA GPU/Orin 部署、BEV 感知模型低延迟推理、Triton 后端优化、离线 engine 构建和吞吐基准测试。
- **优势或局限**：优势是 NVIDIA GPU 性能上限高；局限是需要处理插件、profile、校准数据和不同硬件/驱动版本的兼容性。
- **参考资料链接**：https://docs.nvidia.com/deeplearning/tensorrt/latest/；https://docs.nvidia.com/deeplearning/tensorrt/developer-guide/index.html

- **名称**：TensorRT-LLM / Model Navigator
- **简介**：TensorRT-LLM 面向大模型推理优化，Model Navigator 可自动化模型导出、转换、正确性测试和 profiling。
- **适用场景**：自动驾驶多模态大模型边缘/云端服务、模型导出流水线、候选引擎性能自动比较。
- **优势或局限**：优势是减少手工调参和部署试错；局限是自动化工具仍需与业务输入输出、插件和验收阈值绑定。
- **参考资料链接**：https://nvidia.github.io/TensorRT-LLM/；https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/model_navigator/docs/README.html

✅ 本次抓取完成（页 5/61，维度 1.2.1 TensorRT）

### 1.2.2 TorchScript

- **名称**：TorchScript
- **简介**：PyTorch 的模型序列化和图执行机制，通过 tracing 或 scripting 将模型转换为可保存、可加载的中间表示。
- **适用场景**：旧版 PyTorch 生产链路、C++ LibTorch 集成、需要冻结模型图并减少 Python 依赖的推理系统。
- **优势或局限**：优势是与 PyTorch/LibTorch 集成成熟；局限是 PyTorch 新导出路线更偏向 `torch.export`、ONNX 和 `torch.compile`，复杂 Python 语义支持有限。
- **参考资料链接**：https://docs.pytorch.org/docs/stable/jit.html；https://pytorch.org/tutorials/beginner/Intro_to_TorchScript_tutorial.html

- **名称**：LibTorch
- **简介**：PyTorch C++ 前端和运行时，可加载 TorchScript 模型并嵌入 C++ 推理程序。
- **适用场景**：C++ harness、车端中间件集成、无 Python 运行时的离线评测工具。
- **优势或局限**：优势是便于接入 C++ 工程；局限是模型转换可维护性取决于 TorchScript 覆盖能力。
- **参考资料链接**：https://pytorch.org/cppdocs/；https://pytorch.org/tutorials/advanced/cpp_export.html

✅ 本次抓取完成（页 6/61，维度 1.2.2 TorchScript）

### 1.2.3 OpenVINO

- **名称**：OpenVINO
- **简介**：Intel 推理优化工具套件，支持模型转换、图优化、CPU/iGPU/NPU 推理和模型压缩。
- **适用场景**：Intel CPU/边缘设备部署、非 NVIDIA 硬件性能基线、视觉模型 INT8 压缩、车端辅助计算节点验证。
- **优势或局限**：优势是 CPU 和 Intel 硬件生态优化充分；局限是自动驾驶 3D 感知中的自定义算子和 GPU 专项优化不如 TensorRT 常见。
- **参考资料链接**：https://docs.openvino.ai/；https://docs.openvino.ai/2024/openvino-workflow/model-optimization.html

- **名称**：Post-Training Optimization Tool / NNCF
- **简介**：OpenVINO 生态的模型压缩与量化工具，支持训练后量化和量化感知训练。
- **适用场景**：在精度可控前提下降低 CPU/NPU 推理延迟和内存占用。
- **优势或局限**：优势是压缩流程工具化；局限是量化代表集、校准指标和回归集必须与真实驾驶分布一致。
- **参考资料链接**：https://docs.openvino.ai/2024/openvino-workflow/model-optimization-guide/quantizing-models-post-training.html；https://github.com/openvinotoolkit/nncf

## 1.3 训练工具链

### 1.3.1 分布式训练（Horovod、DDP）

- **名称**：PyTorch DistributedDataParallel（DDP）
- **简介**：PyTorch 原生数据并行训练方案，每个进程绑定一个 GPU，前向独立执行，反向阶段通过 collective 通信同步梯度。
- **适用场景**：多 GPU 感知模型训练、单机多卡和多机多卡 baseline、模型可放入单卡显存但数据量较大的任务。
- **优势或局限**：优势是官方支持、性能稳定、调试资料丰富；局限是超大模型还需要 FSDP、ZeRO 或 pipeline/tensor parallel。
- **参考资料链接**：https://docs.pytorch.org/docs/stable/generated/torch.nn.parallel.DistributedDataParallel.html；https://docs.pytorch.org/tutorials/intermediate/ddp_tutorial.html

- **名称**：Horovod
- **简介**：Uber 开源的分布式训练框架，基于 ring-allreduce 等通信算法，支持 TensorFlow、Keras、PyTorch 和 MXNet。
- **适用场景**：已有 Horovod 训练栈维护、跨框架训练集群、Spark/Kubernetes 上的分布式训练。
- **优势或局限**：优势是多框架统一接口；局限是 PyTorch 原生 DDP/FSDP 生态近年更主流，新项目需评估维护成本。
- **参考资料链接**：https://horovod.readthedocs.io/；https://github.com/horovod/horovod

✅ 本次抓取完成（页 8/61，维度 1.3.1 分布式训练（Horovod、DDP））

### 1.3.2 Mixed Precision / AMP

- **名称**：PyTorch AMP
- **简介**：自动混合精度训练工具，使用 autocast 自动选择低精度算子，并用 GradScaler 缓解 FP16 梯度下溢。
- **适用场景**：视觉感知、BEV、Transformer 训练加速，显存压力较高的多模态模型，训练吞吐优化。
- **优势或局限**：优势是改动小、收益明显；局限是数值敏感损失、稀疏算子或自定义 CUDA op 需要白名单、黑名单和精度回归。
- **参考资料链接**：https://docs.pytorch.org/docs/stable/amp.html；https://docs.nvidia.com/deeplearning/performance/mixed-precision-training/index.html

- **名称**：BF16 / FP8 训练
- **简介**：BF16 提供更大指数范围，FP8 用于更激进的吞吐和显存优化，通常依赖新 GPU 架构和框架支持。
- **适用场景**：大模型预训练、多传感器 Transformer、Hopper/Blackwell 等硬件上的高吞吐训练。
- **优势或局限**：优势是减少显存和带宽压力；局限是硬件、框架和算子覆盖有门槛，FP8 需要更严格的精度监控。
- **参考资料链接**：https://docs.nvidia.com/deeplearning/transformer-engine/user-guide/index.html；https://pytorch.org/blog/accelerating-training-on-nvidia-gpus-with-pytorch-automatic-mixed-precision/

### 1.3.3 调参与训练监控（Weights & Biases, MLflow）

- **名称**：Weights & Biases
- **简介**：实验跟踪、超参搜索、artifact 管理和模型评估平台，可记录指标、配置、可视化样本和模型版本。
- **适用场景**：大规模实验对比、训练稳定性分析、团队共享 dashboard、自动驾驶感知模型 ablation 管理。
- **优势或局限**：优势是交互体验强、可视化丰富；局限是云端合规、成本和离线部署策略需要提前确认。
- **参考资料链接**：https://docs.wandb.ai/；https://docs.wandb.ai/guides/sweeps；https://docs.wandb.ai/guides/artifacts

- **名称**：MLflow Tracking
- **简介**：开源 MLOps 平台的实验跟踪组件，记录参数、指标、artifact 和运行环境，可对接 Model Registry。
- **适用场景**：私有化实验管理、训练流水线元数据追踪、模型版本和指标留痕。
- **优势或局限**：优势是开源、易私有化；局限是高级协作体验和复杂可视化通常需要额外平台建设。
- **参考资料链接**：https://mlflow.org/docs/latest/tracking.html；https://mlflow.org/docs/latest/model-registry.html
