# 3. 模型管理与部署

## 3.1 模型版本管理

### 3.1.1 MLflow

- **名称**：MLflow Tracking
- **简介**：记录参数、指标、工件、代码版本和运行环境，是训练 harness 中最常用的实验追踪层之一。
- **适用场景**：自动驾驶感知模型的多数据集、多配置、多 seed 训练对比；CI 中保存 nightly training 结果。
- **优势或局限**：优势是 API 简洁、后端存储可选、生态成熟；局限是大规模多租户权限和治理能力通常需要额外平台化建设。
- **参考资料链接**：https://mlflow.org/docs/latest/tracking.html

- **名称**：MLflow Model Registry
- **简介**：为模型提供注册、版本、阶段流转、描述和 artifact URI 管理。
- **适用场景**：将训练产物从 candidate、staging 到 production 做审计化流转；管理 ONNX、TensorRT engine、PyTorch checkpoint 的关联关系。
- **优势或局限**：优势是与 Tracking 和 deployment APIs 统一；局限是 registry 只管理元数据，底层 artifact 存储、签名和安全扫描仍需配套。
- **参考资料链接**：https://mlflow.org/docs/latest/model-registry.html

- **名称**：MLflow Models
- **简介**：定义统一模型打包格式和 flavor，支持 Python function、PyTorch、TensorFlow、ONNX 等类型。
- **适用场景**：把训练 harness 产物封装成可复现推理包，供 batch evaluation、Triton 转换或服务化验证使用。
- **优势或局限**：优势是跨框架封装统一；局限是自动驾驶多输入、多传感器预处理逻辑往往需要自定义 wrapper。
- **参考资料链接**：https://mlflow.org/docs/latest/models.html

### 3.1.2 Weights & Biases

- **名称**：W&B Experiments
- **简介**：面向实验追踪、可视化、对比和协作的 MLOps 工具。
- **适用场景**：查看 loss、mAP、NDS、轨迹误差、GPU 利用率、样本可视化和训练曲线。
- **优势或局限**：优势是可视化和协作体验强；局限是企业私有化、数据出境和成本需要提前评估。
- **参考资料链接**：https://docs.wandb.ai/guides/track/

- **名称**：W&B Artifacts
- **简介**：对数据集、模型权重、评估报告等 artifact 做版本化 lineage 管理。
- **适用场景**：追踪某个感知模型版本依赖的数据切片、标注版本、训练代码和评估输出。
- **优势或局限**：优势是 lineage 清晰；局限是超大规模原始传感器数据通常只记录 manifest 或引用外部对象存储。
- **参考资料链接**：https://docs.wandb.ai/guides/artifacts/

- **名称**：W&B Model Registry
- **简介**：在 W&B 中将模型 artifact 注册为可治理资产，支持 alias、审批和使用记录。
- **适用场景**：团队内部共享候选模型，连接训练、评估、部署和回滚流程。
- **优势或局限**：优势是与实验和 artifacts 深度集成；局限是需要统一命名、权限和生命周期规范，否则 registry 容易膨胀。
- **参考资料链接**：https://docs.wandb.ai/guides/model_registry/

## 3.2 模型部署框架

### 3.2.1 NVIDIA Triton

- **名称**：NVIDIA Triton Inference Server
- **简介**：支持 TensorRT、ONNX Runtime、PyTorch、TensorFlow、OpenVINO、Python backend 等多后端推理服务。
- **适用场景**：自动驾驶云端回放评测、数据挖掘批推理、仿真服务和边缘 GPU 服务。
- **优势或局限**：优势是动态 batching、模型并发、metrics 和多框架统一；局限是模型仓库、实例组、显存和延迟配置需要细致调优。
- **参考资料链接**：https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/

- **名称**：Triton Model Repository
- **简介**：通过目录结构和 `config.pbtxt` 描述模型版本、输入输出、backend 和实例配置。
- **适用场景**：管理多版本感知模型、前后处理 ensemble、TensorRT engine 与 ONNX fallback。
- **优势或局限**：优势是部署结构标准化；局限是配置错误会导致动态 shape、batching 或 dtype 问题难排查。
- **参考资料链接**：https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/model_repository.html

- **名称**：Triton Performance Analyzer / Model Analyzer
- **简介**：用于压测吞吐、延迟、并发和模型实例配置的工具链。
- **适用场景**：寻找 perception model 的 batch size、instance count、TensorRT profile 和 GPU 利用率平衡点。
- **优势或局限**：优势是量化调优闭环；局限是压测数据必须接近真实输入尺寸和分布，否则结论偏差很大。
- **参考资料链接**：https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/perf_analyzer/docs/README.html

### 3.2.2 TorchServe

- **名称**：TorchServe
- **简介**：PyTorch 官方生态的模型服务工具，提供 REST/gRPC 推理、模型管理、batching 和 metrics。
- **适用场景**：快速服务化 PyTorch 模型，构建内部评测 API 或 prototype deployment。
- **优势或局限**：优势是 PyTorch 适配直接；局限是项目维护状态和生产长期选型需要结合组织风险评估。
- **参考资料链接**：https://pytorch.org/serve/

- **名称**：Model Archiver
- **简介**：将模型权重、handler、依赖文件打包为 `.mar` 文件。
- **适用场景**：封装 perception 模型的预处理、后处理、类别映射和推理逻辑。
- **优势或局限**：优势是交付单元清晰；局限是复杂多传感器 pipeline 的 handler 容易过重，需要模块化。
- **参考资料链接**：https://pytorch.org/serve/model-archiver.html

- **名称**：TorchServe Metrics
- **简介**：提供系统和模型级指标，并可对接 Prometheus/Grafana。
- **适用场景**：监控推理延迟、请求数、错误率和资源消耗。
- **优势或局限**：优势是服务侧观测开箱即用；局限是业务指标如 mAP 漂移、场景失败率需要自定义埋点。
- **参考资料链接**：https://pytorch.org/serve/metrics.html

### 3.2.3 Kubeflow Serving

- **名称**：KServe
- **简介**：Kubernetes 原生模型服务平台，前身常被称为 Kubeflow Serving，支持 serverless、autoscaling、canary 和多框架 runtime。
- **适用场景**：在 K8s 上部署自动驾驶感知服务、批推理入口、在线模型评估服务。
- **优势或局限**：优势是云原生治理、弹性和流量管理强；局限是依赖 Knative/Istio 等组件时运维复杂度较高。
- **参考资料链接**：https://kserve.github.io/website/

- **名称**：InferenceService
- **简介**：KServe 的核心 CRD，用于声明 predictor、transformer、explainer 等组件。
- **适用场景**：把预处理、模型推理和解释/后处理拆成可独立扩缩容的服务。
- **优势或局限**：优势是资源和路由统一声明；局限是冷启动、镜像体积和 GPU 调度会影响延迟。
- **参考资料链接**：https://kserve.github.io/website/docs/concepts/resources/inferenceservice

- **名称**：Kubeflow Model Registry
- **简介**：集中登记模型、版本和元数据，并与 KServe 等 serving 系统衔接。
- **适用场景**：把训练平台输出的模型登记后再触发部署、验收和灰度流程。
- **优势或局限**：优势是模型元数据标准化；局限是 artifact 仍存储在外部系统，需要治理 URI 可用性和权限。
- **参考资料链接**：https://www.kubeflow.org/docs/components/model-registry/

## 3.3 CI/CD 与自动化训练

### 3.3.1 自动化训练流水线

- **名称**：Kubeflow Pipelines
- **简介**：用于构建、编排和追踪机器学习工作流的 Kubernetes 原生流水线系统。
- **适用场景**：自动执行数据抽样、训练、评估、导出、压测、注册和部署前验收。
- **优势或局限**：优势是组件化和可复现；局限是 pipeline SDK、镜像构建和集群权限需要平台规范。
- **参考资料链接**：https://www.kubeflow.org/docs/components/pipelines/

- **名称**：GitHub Actions / GitLab CI
- **简介**：通用 CI/CD 系统，可触发代码检查、训练 smoke test、模型导出和文档发布。
- **适用场景**：PR 阶段运行 unit tests、小样本训练、ONNX export 和静态检查。
- **优势或局限**：优势是与代码仓库集成紧密；局限是 GPU runner、缓存和大数据访问需要专门配置。
- **参考资料链接**：https://docs.github.com/actions

- **名称**：Argo Workflows
- **简介**：Kubernetes 上的容器工作流引擎，常用于长任务 DAG 和批处理。
- **适用场景**：大规模回放评测、数据挖掘、离线推理和自动化 benchmark。
- **优势或局限**：优势是 DAG 能力强、K8s 生态成熟；局限是 ML 元数据能力需要与 MLflow/W&B/KFP 衔接。
- **参考资料链接**：https://argo-workflows.readthedocs.io/

### 3.3.2 单元/集成测试

- **名称**：Pytest
- **简介**：Python 测试框架，适合覆盖数据处理、模型 wrapper、导出逻辑和评估指标。
- **适用场景**：验证 transforms、collate function、metric 计算、checkpoint load 和 shape contract。
- **优势或局限**：优势是简单灵活、插件丰富；局限是 GPU/e2e 测试需要控制耗时和资源隔离。
- **参考资料链接**：https://docs.pytest.org/

- **名称**：Golden Dataset Test
- **简介**：用固定小数据集和固定期望输出检测模型 pipeline 回归。
- **适用场景**：验证传感器同步、坐标变换、后处理 NMS、tracking ID 稳定性。
- **优势或局限**：优势是回归信号强；局限是 golden 数据必须版本化，并定期审计是否代表真实场景。
- **参考资料链接**：https://martinfowler.com/bliki/RegressionTest.html

- **名称**：Model Export Contract Test
- **简介**：对 PyTorch、ONNX、TensorRT 等不同格式输出做数值一致性检查。
- **适用场景**：部署前验证导出模型与训练模型在固定输入上的误差范围。
- **优势或局限**：优势是能提前发现算子、dtype、动态 shape 和预处理差异；局限是阈值需要按任务和精度策略制定。
- **参考资料链接**：https://onnxruntime.ai/docs/performance/model-optimizations/float16.html

### 3.3.3 模型监控与告警

- **名称**：Online Serving Metrics
- **简介**：监控请求量、延迟、错误率、GPU/CPU 利用率、队列长度和 batch 行为。
- **适用场景**：Triton/KServe/TorchServe 在线服务和离线批推理集群。
- **优势或局限**：优势是能快速定位服务健康问题；局限是不能直接反映感知质量，需要叠加业务评估指标。
- **参考资料链接**：https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/metrics.html

- **名称**：Data Drift / Concept Drift Monitoring
- **简介**：监控输入分布、场景标签、天气、道路类型、目标类别和模型输出分布变化。
- **适用场景**：发现新城市、新传感器版本、长尾场景导致的性能下降。
- **优势或局限**：优势是帮助建立数据闭环；局限是自动驾驶真实质量仍需离线标注评估和场景化 KPI。
- **参考资料链接**：https://www.evidentlyai.com/ml-monitoring

- **名称**：Alert Routing
- **简介**：将服务指标、训练失败、模型质量退化和资源异常转换为告警并路由给责任团队。
- **适用场景**：GPU 集群故障、训练 loss 异常、部署版本错误、推理延迟超阈值。
- **优势或局限**：优势是缩短发现时间；局限是阈值过粗会产生告警疲劳，需要分层和抑制规则。
- **参考资料链接**：https://prometheus.io/docs/alerting/latest/alertmanager/
