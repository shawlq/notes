# SparseDrive 量化策略

本文档总结针对本仓库 SparseDrive 模型（ResNet50 + FPN + 多任务 Head，含 FlashAttention、可选 deformable CUDA、`auto_fp16` 等）的可行量化思路、整体设计、阶段计划与具体操作。

---

## 1. 模型侧：量化难点与敏感点

### 架构要点

- **主干**：`ResNet` + `FPN`，典型卷积堆叠，是 INT8 量化收益最大、也最容易做好的部分。
- **Head**：检测 / 地图 / 运动与规划多分支，含 Transformer 类模块、实例记忆（`InstanceBank`）、时序等；对数值误差更敏感。
- **注意力**：`attention.py` 中 FlashAttention 路径要求 FP16/BF16（如 `q.dtype in [torch.float16, torch.bfloat16]`），不能直接把整条 Attention 换成 INT8 矩阵乘而不改实现。
- **可选算子**：`use_deformable_func=True` 时使用自定义 CUDA（如 `feature_maps_format`），部署工具链往往无现成 INT8 实现，需要 FP16/FP32 保留或自写插件。
- **训练栈**：MMCV/MMDet、动态控制流、字典 `metas`，对 FX 符号追踪 / ONNX 导出不友好，会影响「一键 PTQ」的路径选择。

### 结论

SparseDrive 更适合 **混合精度量化**（Backbone/部分 Conv 做 INT8，Attention/LayerNorm/部分投影与小 Head 保持 FP16），而不是追求全网络 INT8。

---

## 2. 可行且有效的方法（推荐顺序）

### A. 混合精度 PTQ（部署向，性价比高）

- **做法**：ResNet+FPN（及部分 1×1 / 3×3 Conv）做 **per-channel INT8**；Head 中 MatMul 密集的线性层可尝试 INT8，Softmax、LayerNorm、FlashAttn 保持 FP16。
- **工具链**：
  - **TensorRT**：GPU 部署常用，支持 explicit quantization + calibration；自定义 Plugin 需单独处理。
  - **ONNX Runtime QDQ**：若可先导出 ONNX，用 ORT 的 QDQ/QuantFormat 做校准。
- **优点**：无需完整复训，校准集可用 nuScenes 子集。
- **风险**：FlashAttn、自定义 op、动态 shape 可能阻断导出，需要 **子图切分** 或 **替换为 PyTorch 标准 MultiheadAttention（推理专用）** 再导出。

### B. 量化感知训练 QAT（精度要求高的场景）

- 在 FP32 或 BF16 上插入 FakeQuantize，用原训练损失 + 少量 epoch 微调。
- 适合：PTQ 掉点明显时，优先对 **Backbone+FPN** 做 QAT，Head 仍 FP16。
- 实现可选：`torch.ao.quantization`（需处理 MMDet 模块与 ConvBN 融合）、NVIDIA PyTorch Quantization、或 TensorRT 训练扩展（若走 TRT 部署）。

### C. 仅权重量化 Weight-only

- INT8/INT4 权重 + FP16 激活；CNN 上更常见为 RTN、逐通道缩放等。
- 适合：显存/带宽收紧、且可接受专用内核（如 CUTLASS、TensorRT-LLM 风格）时；工程集成成本需单独评估。

### D. 后端原生 FP16/BF16（量化前的基线）

- 推理用 AMP FP16 或 BF16，配合 TensorRT FP16 或 Torch Compile，常能拿到接近 FP32 的精度与明显加速。
- 作为量化前的 **基线**，避免上来就 INT8 折腾一圈还不如 FP16。

## 3. 整体设计（推荐架构）

**设计原则**

1. **划界**：以 **FPN 输出（或多尺度 tensor）** 为界，之前 INT8，之后 FP16；插入 scale/zero-point 或 cast。
2. **算子策略**：
   - Conv / ConvTranspose / 部分 Linear：INT8 优先。
   - LayerNorm / Softmax / GELU / 归一化 / 注意力：FP16。
   - FlashAttn：推理可改为 **标准 MHA + FP16**（略慢但 toolchain 友好），或保留 TensorRT Plugin。
3. **校准**：用真实预处理流水线与评测子集（如 `version='mini'` 或固定帧列表），batch、分辨率与训练一致（如配置中的 `704×256`）。
4. **指标**：分任务看 NDS/mAP、地图指标、运动与规划相关指标；量化不能只盯检测。

**流程示意**

```
多视角图像 → [R50+FPN: INT8 PTQ/QAT] → 特征反量化 FP16 → [Sparse Head: FP16]
                                              → Attention FP16 → Deformable 等 FP16 或插件
```

---

## 4. 量化计划（阶段划分）

| 阶段 | 目标 | 主要动作 |
|------|------|----------|
| **0. 基线** | 可复现 FP16 推理延迟与精度 | 关闭不必要训练逻辑，固定输入尺寸；记录各任务指标 |
| **1. 可导出性** | 明确部署路径 | 尝试 ONNX：必要时简化 forward（单尺度测试图）；处理 `metas`；`use_deformable_func` 试 false 或替换实现对比 |
| **2. PTQ 主干** | R50+FPN INT8 | TensorRT calib 或 ORT QDQ；小校准集（可先试 500～2000 帧量级） |
| **3. Head 渐进** | 控制精度损失 | 先全 FP16 Head；再只对 Linear 试 INT8；Attention 保持 FP16 |
| **4. 不达标则 QAT** | 收回 PTQ 损失 | Backbone+FPN QAT，5～20 epoch 量级试跑 |
| **5. 集成与回归** | 上车/上线条件 | 延迟、内存、各任务指标门禁；边界场景（夜间、遮挡）抽样 |

---

## 5. 具体操作清单

1. **定义推理入口**：单独脚本 `forward` 仅依赖 tensor + 必要 `metas`，避免分散控制流，便于 trace。
2. **FlashAttention**：推理分支改用标准 `nn.MultiheadAttention` 或 mmcv 注意力（FP16），与 Flash 路径对齐数值抽查一层输出差异。
3. **Deformable**：若 TensorRT 无支持，保持 FP16 子图；或暂时 `use_deformable_func=False` 看精度与速度 trade-off。
4. **融合**：部署前 Conv+BN+ReLU 融合（仓库中有 `tools/fuse_conv_bn.py` 可核对是否适用当前计算图）。
5. **校准数据**：从 nuScenes 抽分层抽样（场景、天气、时段）；输入与 `test_pipeline` 一致。
6. **TensorRT 路径顺序**：ONNX → Polygraph 对比 FP32 → build FP16 engine → INT8 calib → 对比输出 tensor / 端到端指标。
7. **敏感性分析**：对 Head 逐模块禁用量化（消融），定位掉点模块，再决定是否 QAT 或永久 FP16。
8. **回归**：复用 `tools/test.py` 与现有 eval hook；记录量化前后同 checkpoint 的对比表。

---

## 6. 结论摘要

- **最有效且现实的路径**：**R50+FPN INT8 PTQ + Head FP16**，必要时对主干做短程 QAT；Attention 与自定义算子不要强行 INT8。
- **首要工程风险** 不在「选 INT8 算法」，而在 FlashAttn / 自定义 op / MMDet 动态图与导出工具链的匹配；量化前应先把 **FP16 TensorRT 或 ONNX Runtime** 跑通作为锚点。
