# 量化评测方法（SparseDrive）

本文整合「量化前后如何评」的通用维度与 **SparseDrive + nuScenes** 下的 **具体执行方式**，便于验收与对比实验。

---

## 一、评测维度总览

量化前后对比，通常从 **精度 / 数值一致性 / 速度与资源** 三条线一起看；研发期可再加 **校准诊断** 与 **难例子集**。

| 维度 | 回答的问题 | SparseDrive 侧要点 |
|------|------------|-------------------|
| 任务级指标 | 量化后任务还能不能用 | 与 `scripts/test.sh` 全链路评测一致，多任务一并报 |
| 输出一致性 | 量化是否贴近浮点模型 | 对比 `results.pkl` 或 hook 关键张量 |
| 校准与数值健康 | PTQ 是否饱和、哪层敏感 | 混合精度：自定义算子与注意力常保留 FP16 |
| 延迟与吞吐 | 快了多少、是否真走 INT8 | `tools/benchmark.py` 或 TensorRT `trtexec` |
| 资源占用 | 体积、显存峰值 | 权重文件大小 + `max_memory_allocated` |
| 鲁棒性（可选） | 长尾场景是否更差 | 子集上重复跑同一套指标 |

**说明**：本仓库训练配置中的 **FP16**（`fp16 = dict(loss_scale=...)`）是训练加速，**不是**部署侧 INT8 量化；量化需在部署栈（PyTorch PTQ/QAT、TensorRT、ORT、芯片 SDK）另行完成。SparseDrive 含 **deformable aggregation** 等自定义 CUDA，整机纯 INT8 往往困难，宜 **分层混合精度**。

---

## 二、各维度如何「具体执行」

### 1. 任务级指标（最重要）

**通用做法**：在 **同一验证集、同一预处理与后处理** 上，分别跑浮点模型与量化模型，对比任务指标；检测/跟踪类需固定 **NMS 与阈值**，否则会放大差异。

**SparseDrive 执行步骤**：

1. 固定：**同一 config**（如 `projects/configs/sparsedrive_small_stage2.py`）、**同一数据 split**（mini 仅冒烟；正式对比用 **完整 val**）、**同一 checkpoint**、**同一随机性**（使用 `--deterministic`）。
2. 运行测试 + 评测（与 `02-性能评测方法.md` 日志一致）：

```bash
bash ./tools/dist_test.sh \
  projects/configs/sparsedrive_small_stage2.py \
  ckpt/sparsedrive_stage2.pth \
  1 --deterministic --eval bbox
```

或使用 `bash scripts/test.sh`（内部调用同上）。

3. **建议记录的指标**（日志中会分段打印）：

| 模块 | 指标 |
|------|------|
| 检测 | mAP、NDS、mATE / mASE / mAOE / mAVE |
| 跟踪 | AMOTA、AMOTP、IDS 等 |
| 地图 | ped_crossing / divider / boundary、mAP_normal |
| 运动 | EPA、min_ade / min_fde、miss_rate（car / pedestrian） |
| 规划 | 各 horizon 的 L2、obj_col、obj_box_col |

4. **对比方式**：表格中列出 baseline 与 quantized 两行，填 **绝对差** 与 **相对变化 %**（如 ΔNDS、ΔL2）。

**注意**：nuScenes **mini** 样本极少，数值会远低于 README 全量 val；对比务必 **同一 split、同一 config**，并与官方 Stage1/Stage2 日志对照时需用完整 val。

---

### 2. 输出一致性（研发期快速定位）

**通用做法**：逐样本对比 logits/特征 **余弦相似度**、输出分布 **KL/JS**、最终向量的 **MAE/MSE**；分类任务可看 softmax 分布偏离。

**SparseDrive 执行步骤**：

- **推荐 A（成本低）**：两次完整推理各保存 **`results.pkl`**（若 `tools/test.py` 支持 `--out`，分别指定 `results_fp32.pkl` 与 `results_int8.pkl`；以当前 argparse 为准）。
  - 按 **同一 `sample_token` 顺序** 解析（可参考 `09-results_pkl解析内容.md`）。
  - 可对比：检测 `boxes_3d`、`scores_3d`、`labels_3d`；规划相关轨迹与 score；全局统计 **预测框数量、score 分布** 是否突变。
- **推荐 B（更细）**：在 `forward` / `head` 上对关键张量 **hook**（如 det 分类 logits、规划轨迹），计算 **余弦相似度、L2、KL**。

**注意**：须固定 **同一套 post_process 与阈值**（配置里 `eval_mode` 的 `tracking_threshold`、`motion_threshhold` 等），否则对比的是流水线差异而非量化误差。

---

### 3. 校准与数值健康（PTQ/QAT 诊断）

**通用做法**：用代表性 **校准集**（数百～数千样本）跑 forward；记录 **每层 scale/zero-point**、激活是否 **大量饱和**；对 loss 敏感层 **保留 FP16 或更高比特**（混合精度）。

**SparseDrive 注意**：

- 校准集可从 nuScenes train 或 val 抽取 **500～2000 帧**，覆盖分布尽量贴近实车（避免只用晴天）。
- 引擎侧：TensorRT 使用 `IInt8Calibrator`；ONNX Runtime 使用相应 calibration；PyTorch 则 `prepare` + 校准数据 `forward`。
- 若 **规划 L2 变差、碰撞率上升**，优先检查规划路径上的模块是否需 **FP16**，而非一味加深 INT8。

---

### 4. 延迟与吞吐

**通用做法**：固定 **batch（常为 1）**、**warmup**，报 **P50/P95/P99** 或平均延迟与 **FPS**；用 profiler 确认 **INT8 路径是否生效**，避免图优化后仍走 FP32。

**SparseDrive 执行**：仓库提供 **`tools/benchmark.py`**，基于真实 dataloader、`samples_per_gpu=1`，预热后计时并打印 **FPS** 与 **GPU 显存峰值**。

```bash
python tools/benchmark.py projects/configs/sparsedrive_small_stage2.py \
  --checkpoint ckpt/sparsedrive_stage2.pth --samples 200
```

量化若在 **TensorRT**，建议用 `trtexec` 或 nsys 查看 **layer time**，核对 INT8 层是否实际执行。

---

### 5. 资源占用

**通用做法**：对比 **模型文件大小**；运行时 **峰值 RSS/显存**；嵌入式可加功耗仪。

**SparseDrive 执行**：`.pth` / ONNX / `.engine` 大小对比；显存可用 `benchmark.py` 中的 `torch.cuda.max_memory_allocated` 思路，量化前后各跑一次取峰值。

---

### 6. 鲁棒性与难例（可选）

**通用做法**：从验证集中划 **难例子集**（夜间、大雨、拥堵、小目标多等），分别统计任务指标。

**SparseDrive 执行**：对子集重复 **第二节 §1** 的全链路评测；规划重点看 **obj_box_col、L2** 是否在难例上恶化更明显。

---

## 三、SparseDrive 量化验收推荐流程（Checklist）

1. **环境锁定**：GPU 驱动、CUDA、PyTorch 或 TensorRT 版本；记录 git commit 与所用 config。
2. **Baseline**：完整 val 上运行 `dist_test.sh` + `--deterministic`，保存日志；可选保存 `results_fp32.pkl`。
3. **量化模型**：按部署栈生成 INT8/混合精度权重或引擎；保证 **输入与 config 一致**（如 6 路相机、`img` 尺寸与预处理）。
4. **重复评测**：同命令再跑，填写 **第二节 §1** 指标表。
5. **一致性抽查**：对同一批 `sample_token` 对比两次 `results_*.pkl`，或抽查 **规划误差大、检测差异大** 的帧。
6. **性能**：运行 `tools/benchmark.py`（或 TRT `trtexec`），对比 **FPS、显存**。
7. **验收阈值（由业务自定示例）**：如 NDS 下降 ≤ 某值、规划 avg L2 上升 ≤ 某值、碰撞率相对 baseline 不恶化超过某值、FPS 提升 ≥ 某值。

**架构提示**：若仅对 **Backbone（ResNet+FPN）** 量化而 **Transformer 头保持 FP16**，仍须做 **全任务评测**（检测/地图/运动/规划），端到端链路会传递数值误差。
